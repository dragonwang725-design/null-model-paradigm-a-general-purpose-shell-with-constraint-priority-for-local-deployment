## 附录A：术语澄清（针对常见误解）

### A.1 "无知识"的精确定义

空模型的"无知识"是指**无后天领域知识（a posteriori knowledge）**，而非**无先验计算能力（a priori computational capacity）**。

| 层级 | 名称 | 归属 | 本质 | 空模型是否"拥有" |
|------|------|------|------|----------------|
| **L0** | 先验计算能力 | 空模型（内置） | 通用算法骨架：BFS、约束满足、路径选择、校验和验证 | ✅ 是（架构常量） |
| **L1** | 元事实（Meta-Facts） | 元事实库（外部） | 静态确定性事实：规则、病历、档案、¬Delete(source_file) | ❌ 否（只读查询） |
| **L2** | 训练知识 | LLM（内部） | 概率性语言模式：互联网语料、文化经验 | ❌ 否（不可见） |

**关键澄清：**
- 空模型查询元事实库，但元事实是**约束条件**，不是空模型的"记忆"
- 类比：CPU 读取内存数据，但数据不是 CPU 的"知识"
- 任务结束后，约束条件丢弃，空模型回到"无记忆"状态

### A.2 "元事实库"替代"隐私资料库"

| 旧称 | 新称 | 说明 |
|------|------|------|
| 隐私资料库（Privacy Vault） | **元事实库（Meta-Fact Vault / Knowledge Base 1）** | 旧称强调"隐私保护"属性；新称强调"人类确认的静态事实层"本质 |

**核心定位：**
- 不是"隐私合规工具"，而是**确定性知识源**
- 隐私保护是元事实的**属性**（因为涉及敏感信息），而非库的**目的**
- 本质功能：约束提取动作、提供因果映射、存储档案

### A.3 "跨领域"的实现机制

因果骨架的"跨领域"不是指代码中不出现领域词汇，而是指**形式推理结构**（A→B→C）跨领域复用。

| 层级 | 内容 | 位置 | 修改方式 |
|------|------|------|---------|
| 元骨架（Meta-Skeleton） | 纯形式节点/边（Node_A→Node_B） | `causal_skeleton.py` | 不可修改（先验计算能力） |
| 领域本体映射 | 实体→形式节点（memory→Node_A） | 元事实库 | 人类维护，按领域配置 |

**代码中的领域名称**（memory_high_usage、ollama 等）是**演示占位符**，实际部署应从元事实库加载映射表。

### A.4 LLM "笼养"的精确机制

LLM 的约束不是"权限剥夺"，而是**"输入边界锁"**：

| 机制 | 说明 |
|------|------|
| 输入边界 | LLM 只能接收空模型过滤后的相关事实 |
| 目的边界 | LLM 策略必须服务于人类注入的目的 |
| 事实边界 | LLM 引用不存在的事实会被标记为"不一致" |
| 元事实盲区 | LLM 不知道元事实库的存在 |

**类比**：LLM 是"封闭考场的考生"——有完整的推理自由，但只能在给定的条件（事实）内作答。

## 核心命题与实现对照（修正后术语体系）

### 命题1：空模型 = 纯粹判断容器，无记忆、无欲望、无目的

| 子命题 | 实现 | 状态 |
|--------|------|------|
| 纯粹判断能力（形式推理、关联、抽象、决策） | `causal_skeleton.py` BFS图距离 | ✅ |
| 无固化内容（不含领域知识、经验、偏见） | **领域知识外置到元事实库（Knowledge Base 1）** | ✅ |
| 无目的意识（无目标、欲望、价值判断） | `purpose_parser.py` 只解析结构 | ✅ |
| 唯一功能：生成可复用判断单元 | `judgment_unit_generator` | ✅ |
| **先验计算能力保留** | BFS、约束满足、校验和验证（L0） | ✅ |

**修正说明：** 原表述"规则外置到配置文件"不够精确。实际架构是**元事实库存储规则**，空模型每次任务前查询，任务后丢弃，不保留任何规则内容。

### 命题2：事实因人而异因目的而异，空模型基于目的提取事实

| 子命题 | 实现 | 状态 |
|--------|------|------|
| 没有绝对的事实，事实取决于角度和目的 | `purpose_parser` + `relevance_engine` | ✅ |
| 空模型以目的为基准提取事实 | `perception_scheduler` 提取前调度 | ✅ |
| 无关目的的信息自动过滤 | `causal_skeleton` 图距离过滤 | ✅ |
| 先无目的接收全量信息，再二次过滤 | 探针全量采集 → 空模型过滤 | ✅ |
| **元事实库约束提取动作** | 查询元事实库获取允许/禁止列表 | ✅ |

### 命题3：LLM只负责策略判断，空模型负责最终判断

| 子命题 | 实现 | 状态 |
|--------|------|------|
| 空模型提取事实，输入LLM做策略判断 | `llm_client.py` 接收脱敏事实 | ✅ |
| LLM返回策略方案 | `llm_client.py` 返回策略 | ✅ |
| 空模型最终判断：是否符合事实和目的 | `final_judgment.py` 校验约束 | ✅ |
| LLM被笼养，零本地执行权 | **输入边界锁机制** | ✅ |
| **LLM不知道元事实库存在** | 架构隔离 | ✅ |

### 命题4：元事实库 = 人类元规则层，决定提取什么

| 子命题 | 实现 | 状态 |
|--------|------|------|
| 元事实库是人类的元规则层 | `meta_fact_vault.py` Rule/Knowledge/Log/MetaFact | ✅ |
| 包含知识、规则、日志、元事实 | 四类数据结构 + **领域本体映射表** | ✅ |
| 空模型在提取前必须查询元规则 | `perception_scheduler.schedule()` | ✅ |
| 隐私不是提取后删除，而是提取动作本身被约束 | **提取前调度** | ✅ |
| **元事实库对LLM完全不可见** | 架构防火墙 | ✅ |
